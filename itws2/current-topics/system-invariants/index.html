1 Objective

The goal of this lecture is

  • to understand the idea of system invariants.
  • to use the Observer pattern to implement system invariants.

2 Example

We consider two objects

View
a DOM element holding a value.
Model
an object holding a state variable x.

2.1 Invariant

The requirement our implementation needs to satisfy is that the view should reflect the model's value.

In other words, the view and model should hold the same value at all times: 

At all time, 
View's value === Model's value

        var tbox = $('#tbox')[0];

4 Model

  • Now we build a 'model' object m.
  • m encapsulates a state variable x.

4.1 Object definition m.

var modelMaker = function(init) {
        var x = init;
    // gets the value of x
        var getX = function() {
                return x;
        }; 

    // sets the value of x to v
    var setX = function(v) {
                x = v;
        };

        var a = {};
        a.getX = getX;
        a.setX = setX;
        return a;
};

var m = modelMaker(5);

4.2 Interaction Interface

m's interface allows for reading and updating its internal state. 

  m.getX : () -> Any
  m.setX : Any  -> Void

5 System Invariant between Model and View

tbox.value === m.getX();

5.1 Implementing the system invariant

5.1.1 Done pushes the value from the view to the model.

var s = function() {
        m.setX(tbox.value); 
        console.assert(tbox.value === m.getX());
};

window.addEventListener('load', function() {
        doneButton.addEventListener('click', s);
        console.log("doneButton now has handler that pushes value to model"); 
});

5.1.2 setX pushes the value from model to view

m.oldSetX = m.setX;

m.setX = function(v) {
        console.log("invoking setX(" + v + ")");

        m.oldSetX(v);
        tbox.value = m.getX();  // push m's value to tbox.

        console.log("tbox.value changed to " + tbox.value);

        g();   // change tbox's edit status

};

6 Observer Pattern

6.1 Problem

The above implementation is not modular!

  • What if many boxes need to be updated after a setX?
  • What if the are not even boxes, but some other objects?
  • How can setX know all the things that need to be done?

6.2 Solution Design

  • have the input element define an onChangeX method.
  • redefine setX so that it runs the onChangeX method of anyone that is registered with it.
  • have the button element register as an observer of the setX 'event'.
  • For each call to setX, the onChangeX methods of each registered observer is run.

6.2.1 define onChangeX

tbox.onChangeX = function(v) {
        tbox.value = v;  // flow v into tbox's value
        g(); // change tbox's edit status
};

6.2.2 redefine m.setX

m.setX = function(v) {
        m.oldSetX(v);
        for (ob of m.observers) {
                ob.onChangeX(v);
        };
}

m.init = function() {
        m.observers = [];
};

m.register = function(ob) {
        m.observers.push(ob);
};

6.2.3 Registering an observer

window.addEventListener('load', function() {
        m.init();
        m.register(tbox);
});

8 Reconciling implementation with requirements

Is the invariant indeed true at all times?

If you carefully observe:

  • In our implementation, the invariant is not true initially. The model m holds the value 5, but tbox's value is empty.

    Exercise: Fix the implementation to ensure the invariant holds at the initial time.

  • The other situation when the invariant is not true is when we are editing the view to enter a new text value.

    It is harder to ensure the invariant during this transient time. The opposite of transient is steady state.

8.1 Weakening the invariant to hold in steady state

The lesson to learn from this is that our original assertion about the invariant is too strong. We weaken it to obtain the following invariant: 

For all times, when the text input box is NOT in edit mode,  
i.e., when the system is in steady state

  tbox.value === m.getX();

8.2 Exercise

Modify the implementation so that tbox2 also is editable. What is the system invariant in this case?
